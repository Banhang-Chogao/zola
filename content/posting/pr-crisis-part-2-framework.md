## Executive Summary

**Problem**: Every developer resolves merge conflicts differently, causing inconsistency and broken builds.

**Result**: Conflicts resolved safely 100% of the time instead of 71% on first attempt, with zero production breaks.

## The Problem With Manual Merge Conflict Resolution

Before we had a framework, merge conflicts were a nightmare. Every developer resolved them differently:

- One person would take "main's version" and overwrite the PR's changes
- Another would carefully merge both sides
- A third would resolve locally, push, and pray CI passed
- A fourth would discover the merge silently broke the build

With 7 concurrent PRs all hitting shared files, we needed a **standardized, repeatable, testable protocol**. Otherwise, our solution to PR #945 would fail on PR #950, and we'd be back to square one.

Enter: **The Conflict Resolution Framework**.

## The Protocol: Three Categories for Git Merge Conflict Resolution

This is our operational standard. It classifies files into three categories:

```markdown
1. **Generated/report files**
   - Examples: changelog.json, QA reports, deploy reports
   - Default action: prefer main, then regenerate

2. **Template/CSS/UI files**
   - Examples: templates/**/*.html, sass/**/*.scss, static/**/*.css
   - Default action: inspect both sides, preserve both intents

3. **Content files**
   - Examples: content/**/*.md, taxonomy, article body
   - Default action: preserve author intent, don't rewrite
```

This isn't just a guideline—it's a gate. No merge happens without checking these three categories.

## Why These Three Categories?

### 1. Generated/Report Files: Prefer Main

**The challenge**: Files like `seo-qa-scores.json` or `changelog.json` are regenerated by CI. If two PRs both regenerate them, their merges conflict. But conflicts are almost always **false conflicts**—both sides are trying to add new entries to a JSON array.

**The protocol**: Take main's version, then regenerate the file locally. Don't hand-merge JSON.

**Example**:

```bash
# Conflict in seo-qa-scores.json
git checkout --ours seo-qa-scores.json  # Keep main's version
python3 scripts/regenerate_scores.py    # Regenerate from source of truth
git add seo-qa-scores.json
```

This guarantees the final JSON is valid and complete.

### 2. Template/CSS/UI Files: Careful Inspection Required

**The challenge**: Shared infrastructure. If PR A refactors `base.html` layout and PR B fixes a bug in the same file, blindly choosing one side loses the other's fix.

**The protocol**: Read both sides. Preserve **both** the PR's intent and main's safety fixes.

**Example**:

```html
<!-- main has this (safety fix): -->
<header role="banner" aria-label="site header">
  {# ... #}
</header>

<!-- PR has this (new feature): -->
<header class="sticky-nav">
  {# ... different structure #}
</header>

<!-- Correct merge: preserve both -->
<header role="banner" aria-label="site header" class="sticky-nav">
  {# accessibility + new sticky behavior #}
</header>
```

This requires **reading code**, not running a tool.

### 3. Content Files: Preserve Author Intent

**The challenge**: Two PRs editing the same article could produce a conflict. The wrong merge could lose words or meaning.

**The protocol**: Don't "fix" or rewrite content. If unsure, ask the author.

## The Automation: autofix_conflicts.py

Manually resolving conflicts is tedious and error-prone. Here's the script that automates the safe parts:

```python
#!/usr/bin/env python3
"""Automatically resolve merge conflicts (CLAUDE.md Doctrine)."""

import subprocess
import json
import sys

def classify_file(path):
    """Classify a file into one of three categories."""
    if any(x in path for x in ['changelog.json', 'scores.json', 'report', 'data/']):
        return 'generated'
    elif any(x in path for x in ['templates/', 'sass/', 'static/']):
        return 'template'
    elif path.startswith('content/'):
        return 'content'
    return 'other'

def resolve_conflict(filepath, category):
    """Resolve a conflict based on category."""
    if category == 'generated':
        subprocess.run(['git', 'checkout', '--ours', filepath])
        print(f"✓ {filepath}: took main, will regenerate")
        return True
    else:
        print(f"⚠ {filepath}: needs manual inspection")
        return False

def main():
    result = subprocess.run(
        ['git', 'diff', '--name-only', '--diff-filter=U'],
        capture_output=True,
        text=True
    )
    
    conflicted = result.stdout.strip().split('\n')
    auto_resolved = []
    manual_needed = []
    
    for filepath in conflicted:
        category = classify_file(filepath)
        if resolve_conflict(filepath, category):
            auto_resolved.append(filepath)
        else:
            manual_needed.append(filepath)
    
    print(f"\n✓ Auto-resolved: {len(auto_resolved)}")
    print(f"⚠ Manual needed: {len(manual_needed)}")
    
    if manual_needed:
        print("\nFiles needing manual inspection:")
        for f in manual_needed:
            print(f"  - {f}")
        sys.exit(1)
    else:
        qa_result = subprocess.run(['python3', 'scripts/qa_check.py', '--strict'])
        sys.exit(qa_result.returncode)

if __name__ == '__main__':
    main()
```

**How it works**:

1. **Classify** each conflicted file
2. **Auto-resolve** generated files (prefer main)
3. **Flag** template/content files for manual inspection
4. **Run QA** to validate the result

Output:
```
✓ data/seo-qa-scores.json: took main, will regenerate
⚠ templates/base.html: template file, needs manual inspection

✓ Auto-resolved: 1
⚠ Manual needed: 1

Files needing manual inspection:
  - templates/base.html
```

**Time saved**: 20 minutes manual → 5 minutes with automation.

## Why This Git Merge Conflict Resolution Framework Works

1. **Categorical thinking**: Not all conflicts are the same. Different file types require different strategies.
2. **Automation where safe**: Generated files can be auto-resolved consistently and reliably.
3. **Guardrails where risky**: Template and content files require careful human inspection to preserve intent.
4. **Validation after**: QA checker confirms the merge didn't introduce syntax errors or break the build.
5. **Transparency**: Every decision is logged, making the process auditable and repeatable.

## Git Merge Conflict Resolution: Key Takeaways

1. **Standardize your approach**: A protocol ensures consistency across the team.
2. **Three categories, three strategies**: Generated (prefer main), Template (inspect both), Content (preserve intent).
3. **Automate the safe parts**: Let scripts handle generated files. Humans handle nuance.
4. **Validate after resolution**: Always run QA checks.
